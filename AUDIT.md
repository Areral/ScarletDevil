# Scarlet Devil Network — Технический аудит и спецификация улучшений

> Дата аудита: 2026-06-04
> Объём: вся кодовая база (Python-оркестратор + Go-ядро + CI + деплой).
> Цель: зафиксировать все проблемы и ошибки, предложить исправления, улучшения, дополнения;
> ответить на вопросы «переписывать ли на C++» и «как увеличить число рабочих прокси в России».

## 0. Краткое резюме (TL;DR)

**Что это.** Агрегатор публичных VPN-конфигов (VLESS/VMess/Trojan/SS/Hysteria2). Каждые 4 часа GitHub Actions
запускает 4 «дрона», каждый: тянет списки-источники, парсит ссылки, прогоняет узлы через Go-ядро
(`angra_core`): L4 TCP-проверка → L7 через `sing-box` (ping + замер скорости) → чемпион-тест топ-5.
Результаты сливаются, дедуплицируются, публикуются как подписки на Vercel + отчёт в Telegram.

**Общая оценка.** Архитектура разумная, код аккуратный и читаемый, но есть **одна фундаментальная
методологическая проблема** и несколько серьёзных багов эффективности.

**Главный вывод по «рабочим прокси в России»:** проверка идёт с раннеров GitHub (США/ЕС), а не из РФ.
«Живой» здесь = «живой со стороны GitHub», что **не равно** «проходит ТСПУ/DPI у российского
провайдера». Это потолок текущей методики — и именно его надо пробивать в первую очередь (см. §5).

**Главный вывод по C++:** переписывать **не нужно** (подробно — §6). Горячий путь уже на Go, и это
правильный инструмент. C++ дал бы 0 выигрыша и большой риск.

**Топ-проблемы по приоритету:**
1. 🔴 Проверка не из РФ → «живость» не отражает реальную доступность за ТСПУ. (§5.1)
2. 🔴 Каждый из 4 дронов парсит ВСЕ источники целиком, потом берёт лишь 1/4 → парсинг/скачивание ×4. (§2.1)
3. 🔴 Нет «памяти» рабочих узлов между запусками: при временном падении источников выдача обнуляется. (§4.1)
4. 🟠 Замер скорости 40 параллельных потоков делят канал раннера → заниженная скорость → хорошие узлы режутся фильтром `min_speed`. (§2.4)
5. 🟠 `git commit --amend` + `git push --force` на `main` каждые 4 часа — хрупко и затирает любые ручные коммиты. (§2.6)

## 1. Архитектура (как есть)

```
GitHub Actions (cron */4h)
 ├─ matrix-crawler  (shard 0..3, параллельно)
 │    main.py
 │      ├─ RKNValidator.load_lists()      # белые списки доменов/CIDR (ТСПУ whitelist)
 │      ├─ LinkParser.fetch_and_parse()   # скачать источники, распарсить ссылки
 │      ├─ [shard slice]                  # взять свою 1/4 узлов
 │      ├─ Inspector.process_all()        # → JSON → go_core/angra_core → JSON
 │      │     L4 (TCP dial) → L7 (sing-box: ping+speed) → champion(top5)
 │      ├─ GeoIP (ip-api.com batch)
 │      └─ Exporter.save_files()          # sub_all_{i}.txt, sub_bs_{i}.txt, sub_chs_{i}.txt, stats_{i}.json
 │
 └─ matrix-merge (needs crawler)
      merge.py
        ├─ собрать stats/sub из артефактов
        ├─ дедуп + слияние → sub_all.txt / sub_bs.txt / sub_chs.txt
        ├─ build index.html
        ├─ Telegram-отчёт
        └─ git commit --amend + push --force   # публикация
            → Vercel раздаёт index.html и /sub*
```

Разделение «Python — оркестрация/парсинг, Go — конкурентные сетевые проверки» **архитектурно верное**.

## 2. Баги и проблемы (с исправлениями)

### 2.1 🔴 Четырёхкратное дублирование парсинга (шардинг сделан не на том уровне)

**Где:** `main.py:31-50`.

Каждый дрон вызывает `fetch_and_parse()` — скачивает и парсит **все** источники полностью, и только потом
берёт срез `all_nodes[shard_index*chunk : ...]`. То есть скачивание, декодирование base64, парсинг,
дедуп и `machine_counts` выполняются 4 раза подряд на 4 раннерах. Шардинг экономит только дорогой этап
L7 (sing-box), а дешёвый-но-не-бесплатный парсинг гоняется ×4. Плюс ×4 нагрузка на источники и на ip-api.

**Исправление (рекомендуемое).** Вынести парсинг в отдельный предварительный job, который один раз
парсит и публикует артефакт `nodes.json`, затем дроны качают его и берут свой срез:

```yaml
jobs:
  parse:
    runs-on: ubuntu-latest
    steps: [ ... python parse_only.py ]   # сохраняет data/nodes.json
    # upload-artifact nodes.json
  matrix-crawler:
    needs: parse
    strategy: { matrix: { shard: [0,1,2,3] } }
    steps:
      - download-artifact nodes.json
      - run: python main.py   # main.py читает nodes.json, НЕ парсит заново
```

В `main.py` — флаг `NODES_FILE`: если задан, читать узлы из файла вместо `fetch_and_parse()`.

**Бонус-исправление шардинга:** сейчас срез `all_nodes[i*chunk:(i+1)*chunk]` отдаёт географически/по-источниково
сгруппированные узлы (узлы одного источника идут подряд) — нагрузка на дроны неравномерна. Лучше
round-robin: `nodes = all_nodes[shard_index::shard_count]`.

### 2.2 🟠 `dead_sources` считается неверно при шардинге

**Где:** `main.py:71-74`.

`m["parsed"]` заполняется при полном парсинге (все источники), а `m["alive"]` — только по своей 1/4 узлов.
Источник, чьи живые узлы попали в другие шарды, ложно помечается «мёртвым» в этом дроне. В `merge.py`
эти множества объединяются, так что итог зашумлён. После фикса §2.1 (узлы шардятся уже после парсинга)
проблема уходит, либо считать `dead_sources` только в parse-job по полному набору.

### 2.3 🟠 GeoIP: повторная резолюция DNS и дубль работы Go-ядра

**Где:** `core/engine.py:158-237`.

- `socket.gethostbyname` через `asyncio.to_thread` запускается для **каждого** узла, без предварительной
  дедупликации хостов → тысячи блокирующих lookups в пул из ~32 потоков. Go-ядро **уже** резолвило IP
  на этапе L4 — резолюция дублируется.
- ip-api.com: бесплатный batch — лимит ~15 запросов/мин; код спит 4.5с между батчами по 100, но это
  делают **все 4 дрона** с близких IP раннеров → риск 429 и флагов `UN`.

**Исправление.**
- Дедуплицировать хосты перед DNS (`set(hosts)`), резолвить уникальные.
- Возвращать IP из Go-ядра в выходном JSON (Go уже его знает) и переиспользовать — DNS в Python убрать.
- Перенести GeoIP в `merge.py` (один проход по финальному дедуплицированному набору вместо 4× по шардам).
- Кэш страны по `/24` подсети, а не по точному IP (соседние IP почти всегда одна страна) — меньше запросов.

### 2.4 🟠 Замер скорости искажён конкуренцией за канал

**Где:** `go_core/main.go:412-431` (`semSpeed = 40`) и `testHTTPSpeed`.

40 параллельных загрузок 512 КБ через 40 прокси делят **один** аплинк раннера (а внутри батча ещё и
несколько батчей по очереди). Измеренная скорость занижена и зашумлена, а `min_speed=1.0 Mbps`
(`go_core/main.go:602`) на этом фоне **выбраковывает живые узлы** → прямая потеря рабочих прокси.

**Исправление (выбрать одно/комбинацию):**
- Разделить «годность» и «ранжирование»: **в выдачу пускать по факту L7-ping** (узел отвечает 204),
  а скорость использовать только для сортировки/метки, не как фильтр отсева. Это самый дешёвый способ
  заметно поднять число прокси.
- Либо снизить параллелизм скоростных тестов (например, 8–12) и/или измерять «time-to-first-byte + загрузка
  64–128 КБ» как прокси-метрику пропускной способности — быстрее и устойчивее к contention.
- Чемпион-тест топ-N оставить последовательным (как сейчас, §`runChampionPhase`) — он даёт честное число
  для витрины.

### 2.5 🟡 Мёртвый код / дубли

- `core/engine.py:370-371` — `Inspector.champion_run()` не используется (чемпион делает Go). Удалить.
- `_is_valid_uuid` продублирован в `core/parser.py:114` и `core/engine.py:20`. Вынести в `core/util.py`.
- Логика «security/tls/reality» размазана по `parser._normalize_config`, `engine._node_to_outbound` и
  `exporter._build_url` — три места знают одни и те же правила. Желателен единый модуль `protocol.py`.

### 2.6 🟠 Публикация через `--amend` + `--force` на `main`

**Где:** `.github/workflows/update.yml:110-121`.

`git commit --amend ... || git commit ...` затем `git push --force` каждые 4 часа переписывает верхушку
истории. Это держит репозиторий маленьким, но:
- любой **ручной коммит** в `main` будет затёрт следующим прогоном бота;
- история фактически отсутствует (нельзя откатиться/сравнить выдачу);
- при гонке/частичном пуше легко получить рассинхрон.

**Исправление (US-007, выполнено).** Публикация в отдельную ветку `gh-pages`:
- `sub_*.txt`, `index.html` и `vercel.json` публикуются обычным коммитом в orphan-ветку `gh-pages`.
- Никаких `--amend` и `--force` против `main` — история `main` не затрагивается.
- Vercel должен быть настроен на деплой из `gh-pages` (Dashboard → Settings → Git → Production Branch).

### 2.7 🟡 Go-бинарь компилируется на каждом прогоне (×4 раннера)

**Где:** `core/engine.py:288-300`.

`angra_core` не закоммичен и не кэшируется в Actions → `go build` на каждом дроне (+`go get`, `go mod tidy`).
`go mod init` к тому же падает (модуль уже есть) — спасает `check=False`, но это шум.

**Исправление.** Собирать бинарь один раз в CI-шаге с `actions/cache` по хэшу `go_core/**`, либо
скомпилировать в parse-job и передать артефактом. Убрать рантайм-`go mod init` (go.mod уже в репо).

### 2.8 🟡 `processSingboxRecursive`: тихая потеря узлов на глубине

**Где:** `go_core/main.go:246-269`.

При повторяющихся «крашах» батч делится пополам до `depth>3` и затем **молча** отбрасывается (`return nil`).
Группы до ~9 узлов могут теряться без следа в статистике. Не критично, но стоит логировать «dropped by
depth limit», чтобы потери были видимы.

### 2.9 🟡 Hysteria2/QUIC проходят L4 безусловно

**Где:** `go_core/main.go:151-155`. Для `hysteria2`/`quic` `checkL4` всегда `true` (UDP не проверяется).
Фактическую достижимость ловит только L7. Это ок как поведение, но для РФ-мобайла UDP/QUIC часто
режется/троттлится ТСПУ — такие узлы «живые на GitHub», но мертвы у пользователя (см. §5).

### 2.10 🟡 Дедуп в `merge_subscription_files` теряет сортировку по скорости

**Где:** `merge.py:149`. `sorted(unique_map.values())` сортирует **строки** ссылок лексикографически,
затирая порядок «по скорости», который выставил `Exporter`. Клиенты обычно пересортируют сами, но если
выдача потребляется «как есть» — лучше сохранять порядок (по первому появлению или по сохранённой скорости).

### 2.11 🟡 `max_accounts_per_server` по `server:port` режет CDN-узлы

**Где:** `core/parser.py:571`, `core/models.py:57`. `machine_id = server:port`. Узлы за одним CDN-доменом
(разные реальные сервера через `host`-заголовок) делят лимит 5 и могут отбрасываться. Если цель —
максимум живых прокси, лимит стоит повысить или ключевать иначе (по `uuid`/`host`+`sni`).

### 2.12 🟢 Мелочи

- `core/settings.py:20` — `BATCH_SIZE` дублирует `checking.batch_size`-смысл; в `engine.py:276`
  читается через `getattr(CONFIG,"BATCH_SIZE",150)`, а конфиг задаёт 100 на верхнем уровне. Источников
  истины два — свести к одному (`checking.batch_size`).
- `core/__init__.py` — пустой комментарий-пасхалка; ок, но проект без `__all__`/версии.
- Нет `.gitignore` для `data/`, `go_core/angra_core*`, `*.log` — артефакты могут случайно попасть в коммит.
- `requirements.txt` без верхней границы у части пакетов закреплён жёстко (`==`) — воспроизводимо, но
  устаревает (aiohttp 3.9.3, pydantic 2.6) — стоит периодически обновлять (есть CVE-фиксы в aiohttp ≥3.10).

## 3. Качество кода и инженерные практики

**Сильные стороны:** аккуратный единый стиль, осмысленные имена, изоляция протоколов в парсере,
устойчивость Go-ядра (recursive split при крахах sing-box), аккуратная работа с base64/URL-кодированием
в экспортере, продуманный GHA-логгер.

**Чего не хватает (по убыванию пользы):**

1. **Тесты.** Нет ни одного. Парсер ссылок — идеальный кандидат на табличные тесты: набор «грязных»
   ссылок VLESS/VMess/SS/Trojan/HY2 → ожидаемый `ProxyConfig`/`None`. Это самая хрупкая часть.
   Добавить `pytest` + корпус фикстур из реальных (анонимизированных) ссылок. Golden-тесты на
   `Exporter._build_url` (round-trip parse→export→parse должен сходиться).
2. **CI-линт.** `ruff` + `mypy` (или `pyright`) на `core/`. Go: `go vet` + `golangci-lint`.
3. **Единый модуль протоколов** (`core/protocol.py`): одно место, описывающее, как
   `ProxyConfig` → sing-box outbound и → URI. Сейчас правила продублированы в 3 файлах и
   расходятся (риск, что L7-конфиг и итоговая ссылка собраны по-разному → узел проверен в одном виде,
   пользователю отдан в другом).
4. **Конфиг как единый источник истины** (Pydantic-модель с типами вместо `Dict[str,Any]` для
   `checking`/`parser`/`app`/`whitelist`) — опечатка в ключе сейчас молча даёт дефолт.
5. **Структурное логирование** ключевых счётчиков в JSON (для дашборда динамики во времени).
6. **README**: добавить раздел «архитектура», «как запустить локально», «какие секреты нужны»
   (`SUBSCRIPTION_SOURCES`, `TG_*`). Сейчас README — маркетинговый, без dev-онбординга.

## 4. Дополнения, повышающие число и стабильность рабочих прокси

### 4.1 🔴 Rolling pool — «память» рабочих узлов между запусками (самый большой выигрыш по количеству)

Сейчас каждый прогон стартует «с нуля» от источников. Если источники временно недоступны или почистились —
выдача проседает/обнуляется. Решение: хранить пул исторически-живых узлов и **перепроверять** его каждый прогон
вместе со свежими из источников.

- После merge сохранять «выжившие» в `pool.json` (узел + кол-во последовательных успехов/провалов + last_seen).
- В начале прогона добавлять к свежим узлам из источников весь пул.
- Узел удаляется из пула после N (например, 3) подряд неудачных проверок (экспоненциальное «забывание»).
- Эффект: выдача перестаёт зависеть от сиюминутной доступности источников; число стабильно-рабочих прокси
  растёт, т.к. накапливаются хорошие узлы, исчезнувшие из источников, но ещё живые.

### 4.2 Расширение и оценка источников

- Вести метрику yield по источнику (parsed → alive%) в `pool.json`, авто-депрайоритизировать «пустые».
- Добавить больше публичных агрегаторов (по аналогии с уже используемыми whitelist-репозиториями).
- Поддержать источники-Clash YAML (не только base64/raw-uri) — там много узлов, которых нет в raw-списках.

### 4.3 UDP-проверка для Hysteria2

Реальный UDP-handshake (или хотя бы успешный L7-ping через hy2-инбаунд) вместо безусловного L4-pass —
отсечёт мёртвые hy2 раньше и честнее их посчитает.

### 4.4 Корректность round-trip

Добавить инвариант: то, что отдаётся пользователю (`Exporter`), должно один-в-один пересобираться обратно
в тот outbound, который проверялся в Go (§3.3). Иначе возможны «проверили рабочий, отдали нерабочий».

## 5. Эффективность в России (ключевой вопрос)

### 5.1 🔴 Фундаментальная проблема: проверка не из РФ

Раннеры GitHub находятся вне РФ. ТСПУ (DPI российских операторов) фильтрует по SNI/протоколу/паттернам
трафика **на пути пользователь→сервер**. Узел, прошедший все проверки с GitHub, может быть наглухо
заблокирован у абонента МТС/Мегафон. Текущая БС/ЧС-классификация (`RKNValidator`) — это лишь *эвристика*
этого: Reality-узлы, чей SNI входит в «мобильный whitelist», с большой вероятностью переживут SNI-фильтр.
Но реальная достижимость из РФ **не измеряется**.

**Что даст наибольший эффект (по убыванию):**

1. **Проба изнутри РФ.** Самый честный способ. Варианты:
   - дешёвый RU-VPS как воркер, который повторяет L7-проверку «выживших» узлов и шлёт вердикт
     (живой/мёртвый из РФ) — публиковать отдельную подписку «RU-verified»;
   - телеметрия от добровольцев (бот/расширение клиента, репортящий успех/неуспех узла из РФ-сетей);
   - даже одна проба из РФ-мобильной сети радикально повысит ценность выдачи.
2. **Эмуляция ТСПУ-условий на стороне чекера** (приближение без RU-VPS):
   - **Reality + whitelisted SNI** считать первым классом (уже частично есть в `check_bs`), но
     дополнительно проверять, что по `dest`/SNI реально отдаётся валидный TLS «легального» сайта
     (Reality-маскировка корректна), а не просто «домен есть в списке».
   - депрайоритизировать/помечать **QUIC/Hysteria2** для мобильного РФ-профиля (UDP/443 часто режется).
   - помечать узлы с «палевными» SNI (домен сервера = IP, самоподписанные, нестандартные порты).
3. **Свежесть и актуальность whitelist'ов.** `check_bs` зависит от внешних списков
   (`config/settings.yaml:42-48`). Если список устарел/недоступен — все узлы падают в ЧС
   (`validator.py:90`). Стоит: кэшировать последнюю успешную версию списков (как rolling pool, §4.1),
   логировать «list age», алертить при пустых базах.

### 5.2 Практические улучшения «под РФ»

- **Класс БС (Nightbird) сейчас = только Reality с whitelisted-SNI.** Это правильная ставка для обхода
  ТСПУ. Усилить: проверять `short_id`/`pbk` валидность (часть уже есть), и что SNI резолвится в РФ-доступный
  адрес.
- **Порты.** Узлы на нестандартных портах живут в РФ хуже; можно метить «443/TCP + Reality» как премиум.
- **Anti-replay / fail2ban на серверах** не наша зона, но стоит метить узлы, отвечающие подозрительно
  одинаково (honeypot-эвристика по идентичным баннерам).

## 6. Стоит ли переписывать на C++? — **Нет.**

**Короткий ответ: нет, это была бы ошибка.**

Разбор по факторам:

| Фактор | Текущий стек (Python+Go) | Гипотетический C++ |
|---|---|---|
| Горячий путь (тысячи конкурентных сетевых проверок) | **Go**: горутины, готовый net/http/SOCKS5, статический бинарь | Нужно вручную городить epoll/потоки, TLS, HTTP, SOCKS — недели работы |
| Бутылочное горлышко | Сетевой I/O + подпроцесс `sing-box`, **не CPU** | C++ не ускоряет сеть/`sing-box` — выигрыш ≈ 0 |
| Парсинг/оркестрация | Python: `pydantic`, `aiohttp`, `base64`, `urllib` «из коробки» | В C++ это медленная разработка и баги парсинга |
| Безопасность памяти | GC, нет UB | ручное управление, риск UAF/overflow на недоверенном вводе из интернета |
| Скорость разработки | высокая | низкая |

**Вывод.** Производительность здесь определяется сетью и внешним `sing-box`, а не языком. Критичная по
скорости часть **уже** на Go — это и есть «правильный C++ для сетевых задач, только безопаснее».
Переписывание на C++ дало бы нулевой прирост производительности при кратном росте сложности и риска.

**Если хочется упростить стек** — направление противоположное: убрать Python↔Go IPC через JSON-файлы и
**слить всё в один Go-бинарь** (Go умеет и HTTP-скачивание источников, и парсинг, и проверки). Это уберёт
двухъязычность, ускорит холодный старт и упростит CI. Но это рефакторинг ради простоты, а не ради скорости,
и делать его не обязательно — текущее разделение рабочее.
