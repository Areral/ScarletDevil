## Codebase Patterns
- **Go settings contract (3 files in sync):** `EngineSettings` struct in
  `go_core/main.go` <- payload built in `core/engine.py` `Inspector.process_all`
  (the `"settings"` dict) <- values read from `core/settings.py CONFIG.checking`
  which loads `config/settings.yaml`. To add a tunable for the Go core: add a JSON
  field on the struct, add the key to the engine.py payload (`CONFIG.checking.get(...)`),
  and document it under `checking:` in settings.yaml.
- **Go build needs `go mod tidy` first:** `go_core/go.sum` may be absent; the build
  fails with "missing go.sum entry" until `cd go_core && go mod tidy` is run (this is
  what engine.py does at runtime on first compile). go.sum should be committed.
- **`go vet` (Go 1.26) runs the hostport analyzer:** `fmt.Sprintf("%s:%d", host, port)`
  passed to net.Dial is flagged. Use `net.JoinHostPort(host, strconv.Itoa(port))`.
- **Local env caveat:** this machine runs Python 3.14 + pydantic 1.x; the repo pins
  pydantic==2.6.1 (uses `field_validator`). `import core.*` fails locally because
  pydantic-core 2.x has no cp314 wheel and source build needs Rust. `py_compile` is
  the reliable local Python check; full imports/pytest validate in CI (3.11/3.12).
- **L7 survival vs ranking:** a node survives the Go L7 phase if it passes the
  connectivity ping (HTTP 204). Speed is measured and stored on every survivor for
  sorting; it only filters when `speed_as_filter: true`.

    before.

  `RKNValidator.load_lists()` + `LinkParser.fetch_and_parse()` and writes
  `data/nodes.json` as a list of `model_dump(by_alias=True)` dicts.
- **Checks:** `python -m py_compile parse_only.py main.py merge.py core/*.py` OK.
- **Learnings for future iterations:**
  - The pre-existing `parse_only.py` (untracked) was already written for this

## Thu Jun  4 - US-005 (GeoIP efficiency: dedup hosts and /24 country cache)
- Deduplicated hosts before DNS resolution and added /24 subnet country cache
  to reduce redundant ip-api lookups (AUDIT ┬¦2.3).
- `core/engine.py`:
  - Added `BatchEngine._geo_cache_key(ip)` static method: returns `/24` network
    string for IPv4 (e.g. `1.2.3.0/24`), exact IP for IPv6. Neighbouring IPs in
    the same /24 almost always share a country, so one ip-api lookup covers the
    whole subnet.
  - Rewrote `Inspector._resolve_geo`: builds a `host -> [nodes]` map first,
    resolves only unique hostnames via DNS, then maps IPs back to all nodes.
    Previously called `gethostbyname` for every node including duplicates.
  - GeoIP fetch list is now deduplicated by /24 cache key (not exact IP) ŌĆö
    after the first IP in a /24 is looked up, subsequent IPs in the same /24
    hit the cache and skip ip-api entirely.
  - Cache store uses `_geo_cache_key(ip_key)` so all IPs in the same /24
    benefit from one lookup.
  - Improved logging: shows `nodes -> unique_hosts`, IP count, cache hits,
    and number of actual ip-api fetches ŌĆö provably quantifies the reduction.
- **Checks:** `python -m py_compile main.py merge.py parse_only.py core/*.py`
  OK. `import core.util` OK. Unit-tested `_geo_cache_key` logic (same-/24
  sharing, cross-/24 isolation, IPv6 exact, garbage fallback). Full
  `import core.engine` fails locally per documented pydantic 1.x/2.x caveat;
  will pass in CI (3.11/3.12).
- **Learnings for future iterations:**
  - `dict.setdefault(key, default)` is the clean Python idiom for building a
    `host -> [nodes]` multi-map in one pass.
  - `ipaddress.IPv4Network(f"{addr}/24", strict=False)` is the correct way to
    get the /24 network from an IPv4 address ŌĆö `strict=False` allows any host
    bits. `str(net)` returns `"x.y.z.0/24"`.
  - The `_GEO_CACHE` is a class-level `Dict[str, str]` on `BatchEngine`,
    shared across all `Inspector` instances within a process. The /24 key
    strategy means once ANY drone resolves one IP in a /24, all subsequent
    nodes in that /24 (even across different batches) reuse the result.
