### Learnings for future iterations:
- **L4 checking живёт в Go, НЕ в Python.** Python только оркестрирует (вызывает Go binary). Если нужно менять параметры проверки — менять нужно в go_core/main.go.
- **IPC между Python и Go — файлы JSON.** Добавление нового канала данных (L4 stats) потребовало третий CLI-аргумент и новый stats-файл.
- **timeout vs connection_refused:** При timeout есть смысл ретраить (сеть может быть перегружена), при connection_refused — нет (сервер активно отвергает).

### Learnings for future iterations:
- **L7 sing-box architecture:** Каждый батч запускает отдельный sing-box процесс с SOCKS-инбаундами.
  Один крашнувшийся outbound может обрушить весь батч → recursive splitting пополам до изоляции проблемного узла.
- **HTTP ping vs speed test:** Два независимых этапа в каждом батче. Ping проверяет connectivity (204 response),
  speed test замеряет пропускную способность (512KB download). Узел должен пройти ОБА этапа.
- **L7 failure categories:** HTTP ping failure происходит через SOCKS-прокси → ошибки могут быть
  timeout (прокси не отвечает), tls_error (TLS handshake failed), bad_status (не 204), http_error (всё остальное).
  Speed test failure добавляет too_slow (недостаточно скорости).
- **Singbox crash ≠ node failure:** Краш sing-box может быть вызван багом sing-box или несовместимой
  конфигурацией, а не проблемой самого узла. Recursive splitting помогает изолировать проблемный узел.
- **Stats channel evolution:** Переход от плоского L4-only stats к вложенному L4+L7 формату.
  Вложенный объект "l7" добавлен без ломки существующего L4 парсинга.
- **Timeout tuning tradeoffs:** Увеличение таймаутов (15s HTTP, 15s speed, 8s startup) улучшает
  detection rate для медленных узлов, но увеличивает общее время проверки. При 50 L4 survivors
  и 50 per batch: ~1 batch → ~35s на батч (8s startup + http checks + speed checks).
